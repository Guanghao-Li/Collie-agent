# Collie-agent 教学文档

这份文档面向想改代码、扩展功能或理解工程实现的人。它会比 [说明文档](说明文档.md) 更接近代码结构，但仍以流程为主，避免逐行解释。

## 1. Runtime 如何装配

应用入口在 `main.py`。常用命令包括 `init`、`run`、`memory` 和 `test-discord`。

真正的系统装配集中在 `bootstrap/app.py`。启动时会创建并连接这些核心对象：

- 配置对象。
- LLM provider。
- `MessageBus` 和 `EventBus`。
- `SessionManager`。
- `MemoryRuntime`。
- `ToolRegistry`。
- `AgentLoop`。
- `ProactiveRuntime`。
- `DriftRuntime`。
- Discord channel。
- Plugin manager。

启动顺序大致是：准备 session 和 memory，加载插件，发送启动事件，连接 Discord，启动被动回复循环，再启动主动推送和闲时任务。

## 2. 被动回复 Pipeline

被动回复的核心在 `agent/loop.py`。

流程可以按下面的顺序理解：

1. Discord 收到消息后进入 `MessageBus`。
2. `AgentLoop` 消费 inbound message。
3. 系统检查是否是命令消息。
4. 普通消息会进入 session 历史。
5. `MemoryRuntime.build_memory_context()` 构建记忆上下文。
6. prompt builder 组装系统提示词、记忆、近期对话和用户输入。
7. LLM provider 生成回复。
8. 如果模型请求工具调用，`ToolRegistry` 执行工具并把结果回填。
9. 最终回复通过 outbound message 发回 Discord。
10. 消息和候选记忆进入 session 与 memory 后续流程。

这个 pipeline 的关键点是：记忆参与提示词，但不会把完整 `HISTORY.md` 注入上下文。

## 3. 记忆系统实现

记忆系统分为运行时、协议层、Markdown 层、旧 store 层和检索层。

核心文件包括：

- `memory/runtime.py`：对外保留旧 public methods，同时初始化新的 `DefaultMemoryEngine`。
- `memory/engine.py`：定义 `MemoryQuery`、`MemoryMutation` 和 `MemoryEngine` 协议。
- `memory/default_engine.py`：把新协议适配到现有 `MemoryStore` 和 `MemorySearch`。
- `memory/markdown_store.py`：管理 Markdown 记忆文件。
- `memory/store.py`：旧结构化记忆 store，继续兼容。
- `memory/search.py`：规则检索。
- `memory/consolidator.py`：把 pending memory 整理进 Markdown 和索引。
- `memory/vector_store.py`：预留向量检索扩展点，默认 disabled。

Markdown 记忆层当前包括：

- `MEMORY.md`：稳定长期记忆。
- `SELF.md`：用户画像、偏好和长期服务规则。
- `HISTORY.md`：按时间追加会话事件摘要。
- `RECENT_CONTEXT.md`：固定结构的近期上下文。
- `PENDING.md`：待整理候选长期记忆。

兼容层继续保留：

- `PROFILE.md`：旧用户画像 mirror。
- `PENDING_MEMORIES.jsonl`：旧 pending memory 输入。
- `MEMORY_INDEX.json`：结构化索引。

### 查询路径

`DefaultMemoryEngine.query()` 默认使用 `MemorySearch` 做规则检索。只有将来显式启用向量记忆并提供真实实现时，才会走向量扩展。

当向量记忆关闭时，行为应与旧系统一致。

### 写入路径

`DefaultMemoryEngine.mutate()` 处理 remember、forget 等 mutation。运行时仍保留旧方法，避免已有调用方崩溃。

consolidation 后会更新 Markdown 层和结构化索引，其中 `HISTORY.md` 是追加写入，不做每轮全量重写。

## 4. 主动推送 Pipeline

主动推送主要在 `proactive/` 下。

核心概念包括：

- Source：产生候选提醒。
- Prefilter：快速过滤明显不适合的候选。
- Judge：判断候选是否值得推送。
- Policy：执行安静时间、每日上限和冷却时间。
- Delivery：通过 `MessageBus` 发出 outbound message。

常见 source 包括：

- `MemoryReminderSource`。
- `RecentContextSource`。
- `ManualCandidateSource`。

主动推送不是直接把候选发出去，而是先经过策略约束和价值判断。这样可以减少打扰感。

## 5. 闲时任务 Pipeline

闲时任务主要在 `drift/` 下。

`DriftRuntime` 会周期性检查用户是否空闲。如果用户最近仍活跃，任务不会执行。达到空闲阈值后，系统会从任务注册表中选择可运行任务。

常见任务包括：

- `MemoryConsolidationTask`：整理候选记忆。
- `RecentContextSummaryTask`：压缩近期上下文。
- `ReflectionTask`：生成反思。
- `ProactiveIdeaTask`：生成主动提醒候选。
- `MemoryDecayTask`：处理可能过期或低价值记忆。

闲时任务的原则是：不要阻塞用户请求，不要在每次对话时做重整理。

## 6. 配置与默认策略

配置通常从 `config.toml` 加载。不同模块只读取自己关心的配置段。

常见配置方向：

- LLM provider 和模型参数。
- Discord 连接参数。
- Memory workspace 和整理策略。
- Proactive 是否启用、安静时间、每日上限、冷却时间。
- Drift 是否启用、空闲阈值、任务频率。
- Vector memory 预留字段。

向量记忆相关字段包括：

- `enable_vector_memory`。
- `vector_db_path`。
- `embedding_model`。
- `vector_score_threshold`。
- `vector_top_k`。

当前默认不启用向量记忆，因此不需要 sqlite_vec 或真实 embedding 服务。

## 7. 扩展点

如果要扩展 Collie-agent，优先从这些位置切入：

- 新工具：扩展 `tools/` 并注册到 `ToolRegistry`。
- 新插件：扩展 `plugins/` 的加载和上下文。
- 新主动来源：实现 proactive source。
- 新闲时任务：实现 drift task 并注册。
- 新记忆后端：实现 `MemoryEngine` 或 `VectorMemoryStore` 协议。
- 新 channel：参考 Discord channel 接入新的消息通道。

扩展时建议保持一个原则：新能力默认可关闭，并且不要破坏旧 public methods。

## 8. 测试建议

文档相关改动通常不需要跑完整测试，但代码改动建议至少覆盖：

- memory runtime 初始化。
- PROFILE fallback。
- query context。
- remember mutation。
- forget mutation。
- 旧 search 和 stats 兼容。
- vector disabled 时 query 行为不变。
- consolidation 后 Markdown 文件内容。
- 主动推送策略边界。
- 闲时任务空闲判断。

常用命令：

```bash
pytest
```

如果只改某一层，可以先跑对应测试文件，再跑全量测试确认没有行为回退。
