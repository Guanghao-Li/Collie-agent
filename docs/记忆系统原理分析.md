# 记忆系统原理分析

## 1. 为什么 AI Agent 需要记忆

普通聊天机器人通常只看当前这一轮输入，或者最多看最近几轮对话。这样会有一个问题：它很难真正“长期陪伴”用户。

例如你告诉它：

```text
我喜欢简洁直接的回答。
```

如果没有长期记忆，过几天你再问问题，它可能已经不知道这个偏好了。

Collie-agent 的记忆系统就是为了解决这个问题。它会把重要信息保存到本地文件里，让 Agent 在未来对话中还能使用这些信息。

## 2. 短期记忆和长期记忆

先区分两个概念：

- 短期记忆：最近一段对话历史。
- 长期记忆：长期稳定、未来仍然有用的信息。

短期记忆像“刚刚聊过什么”。长期记忆像“这个用户长期喜欢什么、正在做什么项目、有哪些目标”。

在代码里：

- 短期记忆由 `SessionManager` 管理。
- 长期记忆由 `MemoryRuntime` 管理。

## 3. 记忆系统保存在哪里

所有长期记忆保存在工作区：

```text
workspace/memory/
```

里面有这些文件：

```text
MEMORY.md
PROFILE.md
RECENT_CONTEXT.md
PENDING_MEMORIES.jsonl
MEMORY_INDEX.json
REFLECTIONS.md
CONSOLIDATION_LOG.md
deleted_memories.jsonl
```

这些文件不是随便分的，而是承担不同职责。

## 4. MEMORY.md：长期稳定记忆

`MEMORY.md` 是给人看的长期记忆摘要。

它保存已经确认、整理、去重后的信息，比如：

```text
- [preference] 用户喜欢简洁直接的回答
- [project] 用户正在构建 Collie-agent
```

它适合人工阅读和快速检查。

## 5. PROFILE.md：用户画像

`PROFILE.md` 更关注“用户是什么样的人”。

它通常包括：

- 用户身份。
- 长期目标。
- 偏好。
- 工作方式。
- 常用项目。
- 明确要求。

专业名词解释：

- “用户画像”：对用户长期特征的总结。不是身份证明，而是帮助 Agent 更好服务用户的上下文。

## 6. RECENT_CONTEXT.md：近期上下文

`RECENT_CONTEXT.md` 保存近期重要情况。

例如：

- 最近在做什么项目。
- 最近有哪些未完成事项。
- 最近对话里有什么重要转折。

它介于短期会话和长期记忆之间。

你可以把它理解成“最近状态快照”。

## 7. PENDING_MEMORIES.jsonl：待整理记忆

不是所有信息都应该立刻写入长期记忆。

例如用户说：

```text
今天我想先看一下这个 bug。
```

这可能只是临时上下文，不一定值得长期保存。

所以 Collie-agent 先把候选记忆写入 `PENDING_MEMORIES.jsonl`。

专业名词解释：

- “pending”：待处理。
- “jsonl”：一种文本格式，每一行都是一个 JSON 对象。它适合追加写入日志型数据。

## 8. MEMORY_INDEX.json：结构化索引

`MEMORY_INDEX.json` 是机器更容易读取的结构化记忆索引。

每条记忆包含：

- `id`：唯一编号。
- `type`：记忆类型。
- `text`：记忆正文。
- `tags`：标签。
- `importance`：重要性。
- `confidence`：置信度。
- `source`：来源。
- `created_at`：创建时间。
- `updated_at`：更新时间。
- `status`：状态。

示例：

```json
{
  "id": "abc",
  "type": "preference",
  "text": "用户喜欢简洁回答",
  "tags": ["communication"],
  "importance": 0.9,
  "confidence": 0.9,
  "source": "command:remember",
  "status": "active"
}
```

专业名词解释：

- “索引”：为了更快查找而整理出来的数据结构。
- “结构化”：数据字段固定，程序容易读取和处理。

## 9. 记忆类型

当前支持这些记忆类型：

- `fact`：事实。
- `preference`：偏好。
- `goal`：目标。
- `project`：项目。
- `relationship`：关系。
- `habit`：习惯。
- `instruction`：明确指令。
- `event`：事件。
- `summary`：摘要。
- `reflection`：反思。

为什么要分类？

因为不同类型的重要性不同。例如“用户长期目标”通常比“一次性闲聊”更重要。

## 10. 记忆状态

记忆状态包括：

- `pending`：待整理。
- `active`：当前有效。
- `deleted`：已软删除。
- `superseded`：被更新内容替代。
- `lowered_confidence`：置信度被降低。

专业名词解释：

- “软删除”：不直接物理删除，而是标记为删除。这样以后还能审计和恢复。
- “置信度”：系统对这条记忆是否可靠的估计。

## 11. 一条记忆如何产生

记忆产生有两条路：

第一条路是用户主动写入：

```text
!remember 我喜欢简洁回答
```

第二条路是系统从对话中自动抽取：

```text
用户：请记住我正在做 Collie-agent 项目。
```

`MemoryExtractor` 会尝试识别这类内容，并生成候选记忆。

## 12. 为什么不直接保存所有对话

如果把所有对话都存成长期记忆，会出现几个问题：

- 记忆太多，搜索结果变差。
- 临时内容污染长期画像。
- 旧信息和新信息冲突。
- 隐私风险更高。
- Agent 可能把一次性想法误认为长期偏好。

所以 Collie-agent 采用“候选记忆 -> 整理 -> 稳定记忆”的流程。

## 13. MemoryExtractor：抽取候选记忆

`MemoryExtractor` 的任务是从一轮对话中找出可能值得保存的内容。

它关注：

- 稳定偏好。
- 长期目标。
- 重要项目。
- 个人资料。
- 明确指令。
- 重要关系。
- 未来需要用到的上下文。

它不应该抽取：

- 临时闲聊。
- 一次性问题。
- 低价值事实。
- 没有证据的猜测。
- 用户没有要求保存的敏感信息。

当前 EchoProvider 下不会调用外部 API，只做简单规则抽取。

## 14. MemoryConsolidator：整理记忆

`MemoryConsolidator` 负责把 pending 记忆整理进长期记忆。

整理流程：

1. 读取 `PENDING_MEMORIES.jsonl`。
2. 读取 `MEMORY_INDEX.json`。
3. 检查重复。
4. 检查冲突。
5. 合并相似记忆。
6. 丢弃低置信度记忆。
7. 写入结构化索引。
8. 渲染 `MEMORY.md` 和 `PROFILE.md`。
9. 写入整理日志。
10. 清空 pending 队列。

专业名词解释：

- “去重”：发现两条意思一样的记忆，只保留或合并一条。
- “冲突”：新旧记忆互相矛盾，例如“喜欢详细回答”和“不喜欢详细回答”。
- “合并”：把重复或相似信息变成一条更完整的信息。

## 15. 冲突如何处理

如果新记忆和旧记忆冲突，系统不会直接删除旧记忆。

它会：

- 降低旧记忆置信度。
- 保留新记忆。
- 在 `CONSOLIDATION_LOG.md` 中记录冲突。

这样做的好处是谨慎。

因为用户可能只是临时改变说法，也可能是真的更新了偏好。直接删除旧信息会丢失历史。

## 16. 搜索是怎么工作的

Collie-agent 第一版不使用向量数据库，而是使用本地打分搜索。

搜索时会综合：

- 关键词匹配。
- 标签匹配。
- 记忆类型加权。
- 重要性。
- 近期性。
- 置信度。

简化公式：

```text
score =
  keyword_score * 0.45
  + tag_score * 0.2
  + importance * 0.2
  + recency_score * 0.1
  + confidence * 0.05
```

专业名词解释：

- “关键词匹配”：查询词和记忆正文有相同词。
- “标签匹配”：查询词和记忆标签匹配。
- “加权”：不同因素的重要程度不同，所以乘以不同系数。
- “近期性”：越新的记忆得分越高。

## 17. 为什么暂时不用向量数据库

向量数据库适合语义搜索，但会带来复杂度：

- 需要 embedding 模型。
- 需要额外存储。
- 需要处理索引更新。
- 测试更复杂。
- 部署成本更高。

Collie-agent 第一版优先结构清晰，所以先用本地关键词搜索。

以后可以在 `memory/search.py` 替换或扩展为向量搜索。

## 18. 记忆如何进入 Agent 回复

一轮普通对话中：

1. 用户发送消息。
2. `AgentLoop` 调用 `MemoryRuntime.build_memory_context()`。
3. 系统搜索相关记忆。
4. 把用户画像、近期上下文、相关记忆加入 prompt。
5. LLM 根据这些上下文生成回复。

也就是说，记忆不是自动“长在模型脑子里”，而是在每次对话前被检索出来，放进模型能看到的上下文。

## 19. 删除记忆为什么用软删除

用户可以通过：

```text
!forget <关键词>
```

删除相关记忆。

系统实际做的是软删除：

- 把 `status` 改成 `deleted`。
- 写入 `deleted_memories.jsonl`。
- 搜索时不再返回这条记忆。

这样既尊重用户的遗忘请求，又保留了审计记录。

## 20. 给技术小白的类比

你可以把记忆系统想象成一个助理的资料柜：

- `PENDING_MEMORIES.jsonl` 像临时便签盒。
- `MemoryConsolidator` 像整理资料的人。
- `MEMORY_INDEX.json` 像电脑里的目录表。
- `MEMORY.md` 像整理好的笔记本。
- `PROFILE.md` 像用户档案。
- `deleted_memories.jsonl` 像碎纸前的登记簿。

Agent 每次回答问题前，会先查资料柜，看有没有和当前问题相关的信息。

## 21. 这个系统的价值

记忆系统让 Collie-agent 具备三个关键能力：

- 连续性：不会每次都像第一次认识用户。
- 个性化：能根据用户偏好调整回答。
- 可审计：记忆保存在本地文件里，人可以检查和修改。

这就是个人 Agent 和普通聊天机器人之间最重要的区别之一。
