# Collie-agent 说明文档

这份文档从功能抽象角度说明 Collie-agent。它不展开太多工程实现细节，主要帮助读者理解系统由哪些能力组成，以及这些能力如何协作。

如果你想了解代码如何落地，请阅读 [教学文档](教学文档.md)。

## 1. 项目定位

Collie-agent 是一个个人助理型 Agent Runtime。它围绕一个长期存在的用户关系运行：接收用户消息，记住长期偏好，在合适的时候主动提醒，并在闲时整理自己知道的内容。

它的核心目标不是做一次性问答，而是让 Agent 在多轮、多天、多场景里保持连续性。

## 2. 能力全景

Collie-agent 可以概括为五类能力：

- 回复：收到用户消息后，根据当前对话、长期记忆和工具结果生成回复。
- 记忆：把稳定事实、用户画像、近期上下文和候选记忆分层保存。
- 主动：在合适的时间，把可能有价值的信息主动推送给用户。
- 闲时：用户不活跃时，做整理、压缩、反思和候选生成。
- 扩展：通过工具和插件接入外部能力。

这些能力不是彼此孤立的。回复会产生新的上下文，记忆会影响回复和主动提醒，闲时任务会整理记忆并生成新的主动候选。

## 3. 被动回复

被动回复指用户先发消息，Agent 再响应。

典型流程是：

1. 用户在 Discord 发送消息。
2. 系统识别会话和用户身份。
3. Agent 读取近期对话和相关记忆。
4. LLM 生成回复，必要时调用工具。
5. 回复发送回 Discord。
6. 新消息和可能的记忆候选被记录下来。

被动回复强调“当下有用”：回答要贴合当前问题，同时利用长期记忆保持连续性。

## 4. 记忆系统

记忆系统使用 Markdown 作为主要可读层：

- `MEMORY.md` 保存稳定长期记忆。
- `SELF.md` 保存用户画像、偏好和长期服务规则。
- `HISTORY.md` 追加会话事件摘要。
- `RECENT_CONTEXT.md` 保存近期压缩上下文。
- `PENDING.md` 保存待整理候选记忆。

系统也保留兼容文件：

- `PROFILE.md` 作为旧用户画像镜像。
- `MEMORY_INDEX.json` 作为结构化索引。
- `PENDING_MEMORIES.jsonl` 作为旧候选记忆来源。

记忆的设计重点是可读、可迁移、可兼容。默认情况下，系统不依赖真实 embedding 服务；向量记忆只是预留扩展点。

## 5. 主动推送

主动推送指 Agent 在用户没有直接发问时，判断某些信息是否值得提醒。

主动推送通常来自三类来源：

- 记忆提醒：长期记忆中有时间、计划或承诺相关内容。
- 近期上下文：最近对话里出现了可能需要跟进的主题。
- 手动候选：外部或插件显式提交了候选提醒。

主动推送不会无限发送。它会受到安静时间、每日上限、冷却时间和候选质量判断限制。目标是“少而准”，而不是刷屏。

## 6. 闲时任务

闲时任务指用户暂时不活跃时执行的后台维护工作。

常见任务包括：

- 整理待处理记忆。
- 压缩近期上下文。
- 生成反思内容。
- 生成主动推送候选。
- 标记可能过期或低价值的记忆。

闲时任务的作用是降低用户交互时的负担。用户提问时，系统尽量快速响应；较重的整理工作放到空闲期完成。

## 7. 配置模型

Collie-agent 的配置可以理解为几组开关：

- LLM：选择模型提供方和调用参数。
- Discord：决定监听和发送到哪里。
- Memory：决定记忆目录、整理策略和检索策略。
- Proactive：决定是否主动推送，以及推送边界。
- Drift：决定闲时任务是否运行，以及何时运行。
- Plugins：决定加载哪些插件和外部能力。

多数能力都应该可以独立关闭。默认策略是先保持安全和稳定，再逐步启用更主动、更复杂的能力。

## 8. 默认边界

Collie-agent 当前遵循这些边界：

- 不强制依赖向量数据库。
- 不强制依赖真实 embedding 服务。
- 不删除旧记忆文件。
- 不把完整历史记录直接塞进回复上下文。
- 不让主动推送绕过频率和安静时间限制。
- 不把闲时任务放到用户请求的关键路径里。

这些边界让系统更容易调试，也更适合渐进式扩展。
