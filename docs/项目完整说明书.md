# Collie-agent 项目完整说明书

## 1. 项目定位

Collie-agent 是一个简化版个人 AI Agent 项目。它的目标不是做一个“大而全”的聊天机器人，而是提供一个可以长期运行、能记住事情、能主动提醒、能在空闲时整理信息、并且可以通过插件扩展能力的个人智能助理框架。

你可以把它理解成一个“住在 Discord 里的个人助理”。用户在 Discord 里和它说话，它会读取最近对话、检索长期记忆、调用工具、生成回复，并在后台定期做主动推送和空闲整理。

第一版重点是清晰、可运行、可测试，所以没有实现 Telegram、QQ、Web 管理后台、多模型路由、MCP、Docker 部署和复杂权限系统。

## 2. 它能做什么

Collie-agent 当前能做这些事情：

- 接入 Discord，接收用户消息并发送回复。
- 支持命令式交互，例如 `!remember`、`!memory`、`!drift`、`!proactive`。
- 保存短期会话历史，让 Agent 在一段对话中知道前面聊过什么。
- 保存长期记忆，把用户偏好、目标、项目、事实等内容写入本地文件。
- 搜索长期记忆，在回答问题时把相关记忆加入提示词。
- 自动抽取候选记忆，把对话中值得长期保存的内容放进待整理区。
- 定期整理记忆，合并重复内容、记录冲突、更新用户画像。
- 主动推送，在合适的时候把可能有用的提醒发到 Discord。
- Drift 空闲任务，在用户长时间没有互动时做低优先级后台工作。
- 注册和调用工具，例如获取时间、安全计算、记忆搜索。
- 通过插件扩展工具、事件处理器、主动推送 source 和 Drift task。
- 使用 EchoProvider 做本地测试，不需要真实 LLM。
- 使用 OpenAI 兼容接口接入真实大模型。
- 用 pytest 覆盖核心模块，不依赖真实 Discord 或真实 LLM API。

## 3. 适合谁使用

这个项目适合三类人：

- 想学习 AI Agent 架构的人。
- 想做一个个人长期助理原型的人。
- 想研究“记忆、主动性、后台思考、插件扩展”这些 Agent 核心能力的人。

如果你只是想要一个普通 Discord 问答机器人，这个项目可能显得结构稍微多一些。它的重点不是“能回复消息”，而是“能长期运行，并逐渐积累上下文和行为能力”。

## 4. 核心模块总览

项目主要由这些模块组成：

```text
main.py
bootstrap/
agent/
bus/
channels/
session/
memory/
proactive/
drift/
tools/
plugins/
plugins_builtin/
tests/
```

每个目录负责一件相对独立的事情。

## 5. main.py：统一入口

`main.py` 是用户启动项目时直接运行的入口。

它只负责：

- 解析命令行参数。
- 加载配置文件。
- 调用 `bootstrap.app.build_app_runtime()` 组装系统。
- 执行对应命令。

它不直接创建所有组件。这样做的好处是：入口文件很薄，项目越长越大时也不会变成一个难以维护的“大杂烩”。

常用命令：

```bash
python main.py init
python main.py run
python main.py memory
python main.py test-discord
```

## 6. AppRuntime：运行时总装器

`AppRuntime` 是整个系统的“总控制台”。

它持有这些核心组件：

- `LLMProvider`
- `MessageBus`
- `EventBus`
- `SessionManager`
- `MemoryRuntime`
- `ToolRegistry`
- `PluginManager`
- `DiscordChannel`
- `AgentLoop`
- `ProactiveRuntime`
- `DriftRuntime`

当你执行 `run` 时，`AppRuntime.start()` 会按顺序初始化组件，并启动后台任务。

当程序关闭时，`AppRuntime.shutdown()` 会停止 Discord、AgentLoop、ProactiveRuntime、DriftRuntime，保存会话，关闭 LLM client，尽量优雅退出。

## 7. DiscordChannel：Discord 交互层

`DiscordChannel` 是 Agent 和 Discord 之间的适配器。

它负责：

- 连接 Discord。
- 监听 Discord 消息。
- 忽略 bot 自己发出的消息。
- 检查允许的用户和频道。
- 把 Discord 消息转换成 `InboundMessage`。
- 从 outbound 队列中读取 Agent 回复。
- 把回复发送回 Discord。
- 把长消息切分成多段，避免超过 Discord 消息长度限制。

为了测试方便，项目支持 fake client。测试不会连接真实 Discord。

## 8. MessageBus：消息总线

`MessageBus` 可以理解成两个异步队列：

- `inbound`：外部消息进入 Agent 的队列。
- `outbound`：Agent 回复外部渠道的队列。

消息流大致是：

```text
Discord 用户消息
  -> DiscordChannel
  -> MessageBus.inbound
  -> AgentLoop
  -> MessageBus.outbound
  -> DiscordChannel
  -> Discord
```

这样做的好处是 AgentLoop 不需要知道 Discord 的细节。以后如果要加别的渠道，也可以复用同一套 AgentLoop。

## 9. EventBus：事件总线

`EventBus` 是一个轻量事件系统。

它允许插件或内部模块监听一些关键时刻，例如：

- 启动前后。
- 一轮对话开始前。
- Prompt 渲染后。
- 调用 LLM 前后。
- 调用工具时。
- 一轮对话结束后。
- 记忆抽取前后。
- 主动推送前后。
- Drift task 前后。

专业名词解释：

- “事件”：系统中发生的一件事，比如“LLM 刚刚返回了回复”。
- “handler”：处理事件的函数，比如收到 `AfterTurnEvent` 后自动抽取记忆。

事件系统的价值是解耦。核心流程不需要写死所有逻辑，插件可以通过监听事件插入自己的行为。

## 10. AgentLoop：对话主循环

`AgentLoop` 是回答用户消息的核心。

它处理一条消息时大致做这些步骤：

1. 从 `MessageBus.inbound` 读取用户消息。
2. 判断是不是内置命令。
3. 如果是命令，直接执行命令并返回。
4. 如果不是命令，读取短期会话历史。
5. 搜索长期记忆，找出相关内容。
6. 构建提示词。
7. 调用 LLM。
8. 检查 LLM 是否请求工具调用。
9. 如果需要工具，调用工具并把结果放回上下文。
10. 最多允许 3 轮工具调用。
11. 保存用户消息和助手回复到短期会话。
12. 抽取候选长期记忆。
13. 把回复发到 `MessageBus.outbound`。

## 11. LLMProvider：模型适配层

项目目前有两个 provider：

- `EchoProvider`
- `OpenAICompatibleProvider`

`EchoProvider` 不调用外部 API，只返回可预测文本，适合测试。

`OpenAICompatibleProvider` 调用 OpenAI 兼容的 `/chat/completions` 接口。

专业名词解释：

- “Provider”：提供某种能力的适配器。这里指“提供 LLM 回复能力”的对象。
- “OpenAI 兼容接口”：虽然服务商不一定是 OpenAI，但接口格式和 OpenAI Chat Completions 类似。

## 12. PromptBuilder：提示词构建器

`PromptBuilder` 负责把各种上下文拼成发给 LLM 的 messages。

一轮对话的提示词包含：

- 系统身份说明。
- 当前时间。
- 相关长期记忆。
- 最近会话历史。
- 可用工具列表。
- 用户当前消息。

专业名词解释：

- “Prompt”：给大模型看的输入说明。
- “System prompt”：系统级指令，用来定义 Agent 的身份、规则和可用上下文。
- “上下文”：模型回答时能看到的信息。

## 13. SessionManager：短期会话

`SessionManager` 负责保存短期对话历史。

每个 session 对应一个 JSON 文件：

```text
workspace/sessions/{session_id}.json
```

短期会话适合保存最近一段对话，比如用户刚刚问了什么、Agent 刚刚回答了什么。它和长期记忆不同，不会默认把所有内容都变成稳定记忆。

## 14. MemoryRuntime：长期记忆系统

`MemoryRuntime` 是项目重点之一。

它不是简单写一个 `MEMORY.md` 文件，而是包含：

- 核心记忆文档。
- 用户画像文档。
- 近期上下文文档。
- 待整理记忆队列。
- 结构化记忆索引。
- 反思记录。
- 整理日志。
- 删除审计记录。

长期记忆文件位于：

```text
workspace/memory/
```

详细原理见：

[记忆系统原理分析](./记忆系统原理分析.md)

## 15. ProactiveRuntime：主动推送系统

`ProactiveRuntime` 让 Agent 不只是被动回答，还能主动检查是否有值得告诉用户的内容。

它会：

- 从多个 source 获取候选内容。
- 根据用户画像、近期上下文和记忆判断相关性。
- 检查 quiet hours。
- 检查每日推送次数上限。
- 避免重复推送。
- 把通过判断的内容发到 Discord。

详细原理见：

[Proactive 原理分析](./Proactive原理分析.md)

## 16. DriftRuntime：空闲后台任务

`DriftRuntime` 用来处理“用户不说话时 Agent 可以做什么”。

例如：

- 整理待处理记忆。
- 总结最近会话。
- 写阶段性反思。
- 生成主动推送候选。
- 衰减长期未使用、低置信度的记忆。

详细原理见：

[Drift 原理分析](./Drift原理分析.md)

## 17. ToolRegistry：工具系统

工具系统允许 Agent 调用一些外部函数，而不只是生成文字。

内置工具包括：

- `get_time`：返回当前时间。
- `calculator`：安全计算基础数学表达式。
- `remember`：写入记忆。
- `search_memory`：搜索记忆。
- `summarize_memory`：总结记忆。
- `trigger_proactive_check`：手动触发主动推送检查。
- `trigger_drift_cycle`：手动触发 Drift cycle。

专业名词解释：

- “Tool”：模型可以调用的函数。
- “Tool call”：模型不是直接回答，而是要求系统帮它执行某个工具。
- “Schema”：工具参数格式说明，比如某个工具需要 `query` 字段。

## 18. 安全计算器

`calculator` 工具没有使用危险的 `eval()`。

它使用 Python 的 `ast` 模块解析表达式，只允许：

- 数字。
- 加减乘除。
- 幂运算。
- 取模。
- 括号。
- 一元正负号。

所以类似下面的输入会被拒绝：

```python
__import__("os").system("rm -rf /")
open("file")
```

## 19. PluginManager：插件系统

插件系统用于扩展 Agent，而不需要改核心代码。

插件可以做四类事情：

- 注册 event handler。
- 注册 tool。
- 注册 proactive source。
- 注册 drift task。

内置插件包括：

- `memory_plugin`
- `proactive_plugin`
- `drift_plugin`

专业名词解释：

- “插件”：外部扩展模块。
- “注册”：把一个功能交给系统管理，让系统知道它存在。
- “Context”：插件 setup 时拿到的一组系统对象，例如配置、工具注册表、记忆运行时等。

## 20. Skill 是什么

用户提到的 “skill” 可以有两种理解。

在这个项目里，目前没有单独实现一个名为 `Skill` 的目录或类。项目使用的是更轻量的插件和工具体系：

- 如果你想让 Agent 学会一个新动作，通常注册一个 tool。
- 如果你想让 Agent 拥有一组能力，通常写一个 plugin。
- 如果你想让 Agent 定期主动检查某类信息，通常写一个 proactive source。
- 如果你想让 Agent 空闲时做某类后台工作，通常写一个 drift task。

因此，在 Collie-agent 当前架构中，skill 更像是“由插件、工具、source、task 组合出来的一项能力”。

举例：

- “记忆 skill” = memory plugin + remember 工具 + search_memory 工具 + 记忆抽取逻辑。
- “主动提醒 skill” = proactive plugin + source + judge + outbound 推送。
- “空闲整理 skill” = drift plugin + task registry + 多个 drift task。

## 21. 配置文件

主要配置在 `config.toml` 中。执行 `python main.py init` 时，项目会把根目录的 `config.example.toml` 复制成 `config.toml`。

现在默认推荐使用单文件配置：Discord、LLM、记忆、主动推送、Drift 和插件配置都放在这一份文件里。`configs/` 目录仍然保留拆分版模板，但只是进阶用法。

也支持环境变量占位符：

```toml
bot_token = "${DISCORD_BOT_TOKEN}"
api_key = "${LLM_API_KEY}"
```

这样你就可以把敏感的 Discord token 和 LLM API key 放到环境变量里，而不是写进配置文件。

示例：

```toml
[app]
name = "Collie-agent"
timezone = "America/New_York"

[discord]
enabled = true
bot_token = "${DISCORD_BOT_TOKEN}"
guild_id = "${DISCORD_GUILD_ID}"
allowed_channel_ids = []
allowed_user_ids = []
default_push_channel_id = "${DISCORD_DEFAULT_PUSH_CHANNEL_ID}"

[llm]
provider = "echo"

[llm.compatible]
model = "${LLM_MODEL}"
api_key = "${LLM_API_KEY}"
base_url = "${LLM_BASE_URL}"
timeout_seconds = 30
temperature = 0.7

[memory]
enabled = true
auto_extract = true
auto_consolidate = true
enable_hyde = true
max_recent_messages = 30
search_limit = 8
workspace_dir = "memory"

[proactive]
enabled = true
interval_seconds = 900
quiet_hours_start = "23:00"
quiet_hours_end = "08:00"
min_score_to_push = 0.72
max_pushes_per_day = 6
fast_prefilter_enabled = true
fast_prefilter_min_score = 0.4

[drift]
enabled = true
interval_seconds = 1800
run_only_when_idle = true
idle_after_seconds = 600
max_tasks_per_cycle = 2

[plugins]
enabled = true
paths = ["plugins_builtin"]
strict_plugins = false
```

## 22. 工作区

工作区是 Agent 保存长期状态的地方。

默认路径：

```text
./workspace
```

里面会有：

```text
workspace/
  sessions/
  memory/
```

代码和状态分开，是为了让你可以升级代码，同时保留用户记忆和会话数据。

## 24. 当前限制

当前版本的限制包括：

- Discord 是唯一交互渠道。
- 没有 Web UI。
- 没有复杂权限系统。
- 没有数据库迁移系统。
- 记忆搜索使用本地关键词打分，不是向量数据库。
- 主动推送判断是简化启发式，不是完整智能决策系统。
- Drift 任务是原型级实现，适合继续扩展。
